## What it does

Drop in a job posting URL or paste the description. The system reads the user's structured profile (a single source of truth: experiences, projects, skills with evidence), reasons about which signal matters most for *this specific role*, and emits a tailored **LaTeX CV** and **cover letter** that compile to a recruiter-ready PDF.

It is, deliberately, **not** a hallucination machine: every claim in the output is grounded in the user's real profile. The LLM picks and orders facts; it does not invent them.

## Why it was hard

CV generation is a deceptively hard LLM problem. The model wants to embellish; the user needs every line to be defensible in an interview. The job posting is noisy and often vague; the system has to extract the *real* signal (what the team actually cares about) and not the boilerplate. And the output is **typeset code** — LaTeX has to compile. A single stray character in a section title kills the whole document.

## System architecture

<Architecture>
{`  job posting (url | text)
       │
       ▼
  ingest        ── extract role, must-haves, nice-to-haves, signals
       │
       ▼
  rank profile  ── score each experience/project against extracted signals
       │
       ▼
  tailor        ── LLM picks ordering, sectioning, emphasis
       │           (output is structured JSON, NOT raw LaTeX)
       ▼
  render        ── deterministic JSON → LaTeX template
       │
       ▼
  compile       ── pdflatex (sandboxed) → PDF
       │
       ▼
  React UI      ── preview + download + per-section regenerate`}
</Architecture>

## How it's built

<Step n={1} label="Profile as the single source of truth">
The user has one structured profile (YAML/JSON): experiences, projects, skills, achievements — each with the *evidence* (dates, links, metrics). Every output traces back to this file. No LLM call gets to invent a new experience.
</Step>

<Step n={2} label="Signal extraction from the posting">
A small ingestion step pulls the job into a structured form: role, seniority, must-have technologies, nice-to-haves, soft signals (team size, domain, growth-stage). This is the only LLM call that touches free-form text from outside the profile.
</Step>

<Step n={3} label="Profile ranking, not generation">
Each profile item is scored against the extracted signals — relevance, recency, evidence strength. The LLM then *picks* which items to include and how to order them, but works only over the structured profile. This is the key design choice: the model is a **selector and reorderer**, never a generator of new claims.
</Step>

<Step n={4} label="Structured output, not raw LaTeX">
The tailoring step emits a JSON schema describing the CV (sections, ordered items, optional emphasis). A deterministic renderer turns that JSON into LaTeX via templates. This is what keeps the output compile-safe even when the model misbehaves.
</Step>

<Step n={5} label="Sandboxed compilation">
`pdflatex` runs in a sandbox with a strict time budget. If compilation fails, the user sees the offending section in the preview and can regenerate just that block — not the whole document.
</Step>

<Failure title="model wanted to invent metrics">
Early prompts gave the model freedom to phrase impact statements. It cheerfully invented numbers ("improved throughput by 40%") that weren't in the profile. The fix: the tailoring step takes a JSON profile in and emits a JSON CV out, with a strict schema that does not allow free-form text outside of fields explicitly marked as `{ source: "profile.experience.X.summary" }`. The LLM cannot phrase a metric that isn't already in the profile.
</Failure>

<Failure title="LaTeX compile fragility">
A single unescaped `&` or `%` from the profile blew up `pdflatex`. The renderer now sanitizes per-field with the right escape rules for the LaTeX context (math mode vs. text vs. URL), and there's a smoke-test that compiles a corpus of "hostile" inputs on every change.
</Failure>

## Why this work matters

This is a small system, but it's a **good shape for any LLM product**: a structured source of truth, an LLM that selects and reorders rather than generates, a deterministic renderer at the boundary, and aggressive failure modes around the model's worst tendencies. That pattern — *use the LLM as the smallest possible part of the pipeline* — is what I keep reaching for when building anything LLM-powered for production.
