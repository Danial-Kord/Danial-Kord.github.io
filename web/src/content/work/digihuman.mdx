## What it does

DigiHuman takes a single RGB camera (laptop webcam, phone) and drives a 3D character in real time — full-body pose, hands, and facial expression. Separately, it can take a single full-body **photo** of a person and produce a **rigged**, animation-ready humanoid mesh of that person — no markers, no studio rig, no manual rigging step.

It is one open-source pipeline that connects four different deep-learning systems and Unity into a single tool that an indie developer or VR researcher can actually use.

## Why it was hard

Each of the four ML pieces — **BlazePose** for body, **MediaPipe FaceMesh** for face, **PIFuHD** for mesh reconstruction, **RigNet** for automatic skinning — is research-quality code with mismatched coordinate systems, mismatched skeletons, mismatched assumptions about the input. None of them ship with a Unity binding. Real-time motion capture also wants smoothness and low latency; research models care about per-frame accuracy and don't.

The interesting part of the project wasn't training a new model — it was the **integration layer**: turning four research pipelines into one real-time system that holds 30+ FPS while a person dances in front of a webcam.

</Architecture>

## How it's built

<Step n={1} label="Pose + face capture (Python)">
A Python service runs MediaPipe's body, hand, and face graphs on each webcam frame, packs the result into a compact JSON payload, and pushes it over a local WebSocket. All ML inference is server-side; Unity is a thin consumer.
</Step>

<Step n={2} label="Skeleton retargeting (Unity, C#)">
BlazePose's 33-keypoint skeleton doesn't match Mecanim's humanoid bone hierarchy. A retargeter converts joint positions into local rotations relative to each parent bone, with a one-time T-pose calibration so the user's proportions don't matter.
</Step>

<Step n={3} label="Smoothing and latency control">
Raw per-frame keypoints jitter. Each joint runs through a **one-euro filter** with a frequency-adaptive cutoff — aggressive smoothing during slow motion, fast response during quick motion. The trade-off knob is exposed in the inspector so animators can tune it for cinematic vs. live-VR feel.
</Step>

<Step n={4} label="Photo-to-rigged-character pipeline">
PIFuHD takes a single full-body photo and returns a watertight 3D mesh — but a raw mesh is useless to Unity without a skeleton. RigNet then predicts joint locations and skinning weights directly from the mesh. Blender headless stitches the two into an importable FBX.
</Step>

<Step n={5} label="Facial expressions">
MediaPipe FaceMesh's 468-vertex mesh is mapped to a fixed set of ARKit-compatible blendshapes (smile, jawOpen, browInnerUp, etc.) using a pre-computed regression. The avatar reads them via Unity's blendshape API.
</Step>

<Failure title="coordinate-system hell">
BlazePose, PIFuHD, and Unity each use a different convention for which axis is up and whether Y or Z grows toward the camera. The first integration silently mirrored the avatar across every frame. The fix was a single explicit `Pose → World` transform applied at the WebSocket boundary — and a unit test that asserts a known T-pose maps to the same humanoid configuration in Unity.
</Failure>

<Failure title="per-joint jitter on the wrists">
Wrists and ankles have the worst keypoint stability because they're often near the edge of the frame and partially occluded. A global smoothing factor either made the body sluggish or left the hands twitching. Per-joint one-euro filter parameters — looser on the spine, tighter on the wrists — was the fix.
</Failure>

## Why this work matters

DigiHuman is an existence proof: it shows that production-grade real-time ML can be glued into a game engine without a render-farm budget. It's also the project that taught me how to write **integration code as a first-class engineering deliverable** — the part of ML systems that almost nobody trains for, and that almost every product needs.
