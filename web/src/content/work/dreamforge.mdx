## What it does

DreamForge turns a sentence — *"a derelict library at the edge of a flooded city"* — into a fully navigable 3D world: rooms, walls with cutouts for windows and doors, props, NPCs, and a camera that frames it cinematically. The runtime is Unity; the brain is a stack of LLMs. The hard parts are not the prompting; they are the **geometry** and the **camera**.

I joined as a senior Unity engineer focused on three pillars: the procedural geometry pipeline, the cinematic camera system, and the CI/CD that lets a small team ship daily.

## Procedural geometry: the wall-cutout system

When the LLM emits a room ("12 × 8 m, two windows on the south wall, an arched doorway"), something has to turn that into actual triangles a player can walk around. I built a **wall-cutout system** using **CSG boolean subtraction** with four interchangeable silhouette strategies, so each wall opening — square window, arched door, irregular hole — uses the right algorithm for its shape:

<Step n={1} label="Convex hull">
For simple shapes (square windows, rectangular doors). Fast, predictable, no edge cases.
</Step>

<Step n={2} label="Radial ray casting">
For shapes with a single clear center (round windows, portholes). Robust to noisy input from the LLM.
</Step>

<Step n={3} label="Voxel boundary tracing">
For arbitrary blob shapes. Slower but always produces a closed loop.
</Step>

<Step n={4} label="Alpha-shape concave hull">
For complex non-convex outlines (gothic arches, broken-wall holes). Handles concavities the convex hull silently destroys.
</Step>

After silhouette extraction, a **BFS connected-component filter** drops orphaned islands, then **ear-clipping triangulation** produces a clean triangle list, and the result is extruded to give the wall thickness.

<Architecture>
{`  LLM prompt
      │
      ▼
  room descriptor  (size, openings[], style)
      │
      ▼
  wall mesh ──┐
              ├── for each opening:
              │       silhouette strategy ──► loop
              │       BFS island filter   ──► clean loop
              │       ear-clipping        ──► triangles
              │       extrude depth       ──► volume
              ▼
  CSG subtract → final wall mesh → collider`}
</Architecture>

## Cinematic camera system

A procedural world is only as good as its camera. Built on **Cinemachine** with several layers stacked on top:

- **Multi-candidate selection.** For any framing target (player + NPC, group of objects, action focus), the system samples N candidate camera positions, scores each by composition and occlusion, and picks the best.
- **3-ray spherecast occlusion fading.** When geometry intrudes between camera and subject, three offset spherecasts probe the volume and the offending mesh fades out instead of clipping or jumping the camera.
- **Bone-based NPC framing.** Camera composition references humanoid bones (head, chest), not pivots, so a tall NPC and a short NPC get framed correctly.
- **Velocity-damped yaw constraints.** Camera yaw is constrained by room geometry; the constraint is damped against player velocity so the camera doesn't snap when the player turns quickly.

<Failure title="silhouette strategy didn't generalize">
The first version used only convex-hull silhouettes — fast and clean for windows, but it silently filled in any concave opening (the gothic arches turned into closed shapes). The fix was making strategy selection a per-opening property emitted by the LLM, with a fallback heuristic that detects concavity from the input loop and switches to alpha-shape automatically.
</Failure>

## Polygon-based room shapes

Rooms aren't always boxes. I built a polygon-based room shape system with:

- **Grid-based flood fill validation** — confirms a generated room polygon encloses a navigable interior before geometry generation runs.
- **Geometry utilities** — segment intersection, polygon scaling, convex hull — all unit-tested, all reusable across the wall, floor, and prop-placement systems.

## CI/CD: Docker + Azure + Slack

Procedural world-gen has a long failure tail (LLM emits an invalid descriptor, geometry pass produces a non-manifold mesh, etc.). The CI/CD pipeline catches these without slowing the team:

- **Docker** containers run headless Unity test passes on each PR.
- **Azure Container Instances** spin up build agents on demand.
- **GitHub Actions** orchestrates: lint → test → build → publish.
- **Slack webhook** posts exception summaries with stack traces and a one-click "create PR with auto-fix" path for known classes of failure.

## What this taught me

LLM-driven systems live or die on the **last-mile classical engineering** between the model output and the user. The model can hallucinate a beautiful room description; the geometry pipeline still has to produce a closed mesh. DreamForge is, more than anything, a story about hardening the pipe between an unreliable generator and a real-time renderer that has zero tolerance for malformed input.
